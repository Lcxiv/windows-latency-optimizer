1 UDP Game State Packets

Why UDP, Not TCP

Competitive shooters use UDP because it has no head-of-line blocking. When a TCP packet is lost, every subsequent packet queues behind the retransmission—even packets carrying newer, more relevant game state. UDP lets the game engine decide what to do with late or lost data. The UDP header itself is only 8 bytes versus TCP's typical 32 bytes, reducing per-packet overhead.

Fortnite's Network Architecture

Fortnite is built on Unreal Engine and uses a client-server authoritative model running at 30 ticks per second (30 Hz server update rate). Historically it launched at 8 Hz and was upgraded. The client update rate is approximately 75 updates/second. Fortnite uses UDP ports 9000–9100 for gameplay traffic, with TCP on ports 80, 443, and 5222 for authentication, matchmaking, chat (XMPP), and Epic Online Services.

Packet Structure (Unreal Engine Internals)

Unreal Engine's UDP packets use a custom reliability layer built on top of raw UDP:

  • Packet Header: Contains a sequence number (monotonically increasing), an ack field (most recent received sequence), and an ack_bits bitfield (32-bit mask encoding which of the last 32 packets were received). This design from Gaffer on Games means each ack is redundantly sent ~32 times across subsequent packets.
  • Bunches: Each packet contains one or more "bunches"—data containers targeted at specific replication channels (actor state, RPC calls, property updates). Bunches can be partial/fragmented across packets.
  • Channels: Each replicated actor gets its own channel. Position, rotation, velocity, and health are replicated continuously. Critical events (damage, eliminations) use a reliable-ordered sub-channel built atop the unreliable UDP transport.

Typical Packet Characteristics

PropertyTypical ValueNotes
MTU / Max Packet Size512–1024 bytesUE defaults to 1024, was lowered to 512 at times to avoid fragmentation
Payload (no IP/UDP headers)100–500 bytes typicalPosition updates are small; big state changes are larger
Send Rate (client→server)~75 packets/secPlayer input (movement, aim, actions)
Send Rate (server→client)~30 packets/secWorld state snapshots at tick rate
IP FragmentationAvoided by designUE keeps packets under 576 bytes when possible (RFC 791 safe)

What Happens When a Packet Is Lost vs Delayed

  • Lost position update: The client continues predicting using the last known velocity. When the next packet arrives, the client interpolates to the corrected position. If the prediction was wrong, you see a small "snap" or rubber-band.
  • Lost critical event (damage/elimination): Sent via reliable-ordered channel. UE's ack_bits detect the miss and retransmit on the next packet. Delay equals one round-trip (~30–60ms at 15–30ms ping).
  • Delayed packet: If a packet arrives with an older sequence number than the most recent, it is discarded. Stale state is worse than no state—the client's prediction from newer data is already more accurate.

Fortnite's Compensation Techniques

  • Client-side prediction: Your own movement is predicted instantly. The server validates and sends corrections. You never feel your own latency for movement.
  • Entity interpolation: Other players are rendered ~50ms in the past (interpolation buffer of ~2 server ticks). This smooths out jitter but means you always see enemies slightly behind their real position.
  • Server-side rewind: For hit detection, the server "rewinds" other players to where they were when you fired (accounting for your ping). This is why high-ping players can sometimes get hits that feel unfair to low-ping players.

2 TCP Retransmissions and Their Gaming Impact

Where TCP Is Used in Games

Even UDP-based games use TCP for non-realtime traffic: matchmaking (HTTPS/443), authentication, friend lists, inventory/item shop, voice chat signaling (XMPP/5222), and telemetry. A TCP stall here won't cause rubber-banding, but it can cause:

  • Slow matchmaking queue joins (frozen "Connecting..." screens)
  • Delayed friend list/party updates
  • Chat message delays
  • Failed item purchases or inventory loads

The Exact Latency Penalty

TCP has two retransmission mechanisms, and their latency penalties differ dramatically:

MechanismTriggerLatency AddedPlayer Experience
Fast Retransmit 3 duplicate ACKs received ~1 RTT (30–60ms typical) Brief hiccup, usually unnoticed
RTO Timeout No ACK received within RTO timer 200–300ms minimum (Windows min RTO = 300ms, Linux = 200ms) Noticeable freeze/stall
Double Loss (RTO + backoff) Retransmitted packet also lost 600ms–1200ms (exponential backoff doubles RTO) Visible hang, potentially disconnected
Initial SYN loss First connection packet lost 3 seconds (initial RTO per RFC 6298) Game takes forever to "connect"
Why Windows RTO is 300ms, not lower: Per Microsoft KB, Windows TCP has a minimum RTO of 300ms. This is conservative by design. Even on a 10ms-ping LAN connection, a single lost TCP packet costs at minimum 300ms. This is why game engines use UDP for gameplay—they cannot afford this floor.

Fast Retransmit vs RTO: What to Watch For

Fast retransmit (triggered by 3 duplicate ACKs) recovers in ~1 RTT because the sender infers the loss from out-of-order ACKs. This is the "good" path. RTO timeout is the "bad" path—it means even the duplicate ACKs were lost, so the sender has no signal and must wait for the full timer to expire. After each RTO, the timer doubles (exponential backoff), so consecutive losses escalate rapidly: 300ms → 600ms → 1200ms → 2400ms.

tshark filters for TCP retransmission analysis: 
# Show all retransmissions
tshark -r capture.pcapng -Y "tcp.analysis.retransmission"

# Show fast retransmissions specifically
tshark -r capture.pcapng -Y "tcp.analysis.fast_retransmission"

# Show RTO-based retransmissions (retransmissions that are NOT fast)
tshark -r capture.pcapng -Y "tcp.analysis.retransmission and !tcp.analysis.fast_retransmission"

# Count retransmissions per conversation
tshark -r capture.pcapng -Y "tcp.analysis.retransmission" -z conv,tcp

# Show duplicate ACKs (precursor to fast retransmit)
tshark -r capture.pcapng -Y "tcp.analysis.duplicate_ack"

# Show zero-window events (receiver overwhelmed)
tshark -r capture.pcapng -Y "tcp.analysis.zero_window"

# Full TCP conversation statistics including retransmissions
tshark -r capture.pcapng -q -z tcp,tree

Three Types of Loss and Their Signatures

TypeCauseCapture SignatureGaming Impact
Random Loss Bit errors (rare on wired), marginal signal (Wi-Fi), minor congestion Evenly distributed missing sequence numbers. No pattern in timing. Low loss rate (<1%). Occasional micro-stutters. Client prediction handles it well. Least harmful type.
Burst Loss Buffer overflow, switch/router queue full, Wi-Fi interference burst, NIC driver bug Clusters of consecutive missing sequence numbers. Gap in frame.time_delta followed by burst of arrivals. Multiple packets lost in <100ms window. Most damaging for games. Multiple state updates lost at once. Client prediction diverges significantly. Causes visible rubber-banding and position snaps.
Congestion Loss Sustained link saturation (someone streaming/downloading), bufferbloat Gradual RTT increase visible in TCP streams. Increasing frame.time_delta variation. Loss Rate Thresholds for Competitive Gaming

Loss RateClassificationPlayer Experience
< 0.1%ExcellentIndistinguishable from zero loss. Tournament-grade.
0.1% – 0.5%GoodRare micro-corrections. Acceptable for competitive ranked play.
0.5% – 1%NoticeableOccasional hiccups. Hit registration feels inconsistent. You lose some gunfights you should win.
1% – 2%ProblematicRegular rubber-banding. Enemies teleport occasionally. Frustrating in build fights.

Critical insight: Burst loss of 2% is far worse than random loss of 2%. Five consecutive lost packets (a 167ms gap at 30 Hz) forces a much larger prediction correction than five packets lost randomly across 10 seconds. Packet captures can distinguish these patterns by examining the distribution of inter-arrival times, not just the total loss count.

4 Jitter and Its Effect on Game Smoothness

What Jitter Actually Is

Jitter is the variation in packet inter-arrival times. If a game server sends packets every 33ms (30 Hz), but they arrive at your client with intervals of 20ms, 45ms, 28ms, 50ms, 15ms—that variation is jitter. It is mathematically defined per RFC 3550 as:

D(i,j) = (Rj - Ri) - (Sj - Si)

Where:
  Rj, Ri = receive timestamps of packets j and i
  Sj, Si = send timestamps of packets j and i
  D(i,j) = the instantaneous jitter sample

Running mean:  J(i) = J(i-1) + (|D(i,j)| - J(i-1)) / 16

Why Low Ping + High Jitter Feels Worse Than Higher Ping + Low Jitter

Consider two connections:

ConnectionAverage PingJitterPacket Arrival Pattern
A: Low ping, high jitter 20ms 40ms Arrivals at: 5ms, 55ms, 12ms, 48ms, 8ms, 60ms...
B: Higher ping, low jitter 40ms 3ms Arrivals at: 38ms, 41ms, 39ms, 42ms, 40ms, 39ms...

Connection B feels dramatically smoother because:

  • The interpolation buffer can be small (the game knows packets arrive on schedule).
  • Client-side prediction rarely needs correction.
  • Enemy movement appears fluid—each frame has fresh data arriving predictably.

Connection A causes:

  • The game must use a larger interpolation buffer (~50ms+) to absorb jitter spikes, adding visual delay.
  • Some frames have two updates to process (when a late packet and the next one arrive together), others have none, creating micro-stuttering.
  • The 60ms spike is effectively a momentary loss event from the game's perspective.

Jitter Thresholds

JitterClassificationEffect in Competitive FPS
< 5msExcellentImperceptible. Esports/tournament-grade.
5–15msGoodSmooth gameplay. Acceptable for ranked competitive.
15–30msNoticeableOccasional micro-stutters. Enemy movement sometimes appears jerky. Hit reg feels slightly off.
30–50msProblematicVisible stuttering. Build fights feel inconsistent. Edits sometimes fail to register.
> 50msSevereEnemies teleport. Game is essentially unplayable competitively.

# Calculate jitter: compute standard deviation of frame.time_delta values
# Expected delta at 30Hz = 0.033s. Jitter = variation around that mean.

5 Bufferbloat Detection in Packet Captures

What Bufferbloat Is

Bufferbloat occurs when oversized buffers in routers, switches, or your ISP's equipment absorb packets instead of dropping them during congestion. The packets eventually arrive—but hundreds of milliseconds late. The result: your ping test shows 5ms when idle, but jumps to 50–200ms+ when someone on your network is streaming or downloading.

The Signature in Packet Captures

Bufferbloat has a distinctive pattern in pcapng files:

  1. Idle baseline: TCP handshake RTT shows true propagation delay (e.g., 5ms to game server).
  2. Under load: RTT climbs linearly as buffers fill. The delay increases proportionally to the queue depth, not the distance.
  3. Steady-state bloat: RTT plateaus at a high value (the buffer is full). This plateau equals buffer_size_bytes / link_rate_bytes_per_sec.
  4. No packet loss during the RTT increase: This is the key differentiator. With proper queue management (like fq_codel), you'd see some loss but stable latency. With bufferbloat, you see zero loss but massive latency inflation.

Measuring Severity

Idle RTTLoaded RTTBloat FactorGaming Impact
5ms15ms3x (minimal)Acceptable. Modern AQM or small buffers. +10ms under load is tolerable.
5ms50ms10x (moderate)Noticeable during concurrent traffic. Your 5ms ping becomes 50ms when someone starts a download.
5ms200ms40x (severe)Unplayable during any concurrent network use. A YouTube video on another device ruins your game.
5ms500ms+100x (extreme)Common on DSL/cable with stock router firmware. Game is unplayable whenever any other device is active.
The deceptive nature of bufferbloat: Standard speed tests show great results (full bandwidth utilized). Standard ping tests (idle network) show low latency. The problem only appears under concurrent load—which is exactly when you're gaming while someone else is using the network.

Detection Method from Packet Timestamps

To detect bufferbloat from a pcapng, you measure RTT during idle periods vs. during high-throughput periods:

tshark approach for bufferbloat detection: 
# Step 1: Capture during a bufferbloat test
# Start a large download while gaming, capture both streams
tshark -i "Ethernet" -w bufferbloat_test.pcapng -a duration:60

# Step 2: Measure TCP RTT via SYN/SYN-ACK handshakes (baseline)
tshark -r bufferbloat_test.pcapng -Y "tcp.flags.syn==1 and tcp.flags.ack==0" \
  -T fields -e frame.time_epoch -e ip.dst

# Step 3: Track RTT over time using TCP ACK analysis
tshark -r bufferbloat_test.pcapng -Y "tcp.analysis.ack_rtt" \
  -T fields -e frame.time_epoch -e tcp.analysis.ack_rtt -e ip.dst

# Step 4: Compare game server RTT during idle vs loaded periods
# Look for: RTT values climbing from baseline (e.g., 5ms -> 50ms+)
# while throughput on other connections is high

# Step 5: Use tcp.time_delta for per-stream timing
tshark -r bufferbloat_test.pcapng -Y "tcp.stream eq N" \
  -T fields -e frame.time_epoch -e tcp.time_delta

# Bloat factor = max(ack_rtt) / min(ack_rtt) for same destination
# Factor > 5x = bufferbloat present

6 Nagle Algorithm Visibility in Packet Captures

How Nagle Works

Nagle's algorithm (RFC 896) prevents sending small TCP segments when there's outstanding unacknowledged data. The logic: if there's unacked data AND the new data is smaller than MSS (Maximum Segment Size, ~1460 bytes), buffer it until either an ACK arrives or enough data accumulates to fill an MSS.

The Deadly Interaction with Delayed ACK

The real problem isn't Nagle alone—it's Nagle + Delayed ACK. The Delayed ACK timer (default 40ms on Linux, 200ms on Windows) causes the receiver to wait before sending an ACK, hoping to piggyback it on a response. Combined with Nagle:

  1. Sender sends small packet #1 (e.g., player input, 50 bytes).
  2. Sender wants to send packet #2 (another input update, 50 bytes).
  3. Nagle says: "Wait, packet #1 isn't ACKed yet, and this is a small packet. Buffer it."
  4. Receiver says: "I'll delay my ACK for up to 40ms hoping to piggyback."
  5. Result: Packet #2 is delayed by up to 40ms (Linux) or 200ms (Windows) waiting for the ACK that unlocks it.

What It Looks Like in a Capture

The telltale pattern:

  • Small packet sent (e.g., 50–200 bytes).
  • Exactly 40ms gap (Linux) or 200ms gap (Windows) before the next small packet.
  • The ACK arrives, then the buffered packet is immediately sent.
  • This pattern repeats rhythmically—send, wait, ACK, send, wait, ACK.
Why this is devastating for games: Games that use TCP for any real-time-ish traffic (position updates, chat, ability cooldowns) and forget to set TCP_NODELAY get a fixed 40–200ms penalty on every other small packet. This is additive with network latency. A 20ms connection feels like 60–220ms.

The Fix: TCP_NODELAY

Setting TCP_NODELAY (socket option) disables Nagle entirely. Every send() call flushes immediately regardless of outstanding ACKs. You can verify whether a connection has it enabled:

tshark detection of Nagle delays: 
# Method 1: Find small packets with suspiciously consistent ~40ms gaps
tshark -r capture.pcapng -Y "tcp.len > 0 and tcp.len < 200" \
  -T fields -e frame.time_delta -e tcp.len -e tcp.stream

# Look for frame.time_delta values clustering around 0.040 (40ms)
# or 0.200 (200ms on Windows). This is the Nagle+DelayedACK signature.

# Method 2: Correlate small sends with ACK arrivals
tshark -r capture.pcapng -Y "tcp.stream eq N" \
  -T fields -e frame.time_delta -e tcp.len -e tcp.flags.ack -e tcp.ack

# Method 3: On a live Linux system, check if TCP_NODELAY is set
# ss -tin state established | grep nodelay

# Pattern to look for:
#   [small pkt]  delta=0.001  len=87
#   [ACK only]   delta=0.040  len=0    <-- delayed ACK after 40ms
#   [small pkt]  delta=0.001  len=92   <-- Nagle-buffered packet released
#   [ACK only]   delta=0.040  len=0    <-- another delayed ACK

7 DNS Resolution at Game Launch

Why Games Take 10–30 Seconds to Connect

When you launch Fortnite and click "Play," the client performs a cascade of DNS lookups before you ever reach a game server:

  1. Authentication: account-public-service-prod.ol.epicgames.com (login tokens)
  2. Friends/Party: xmpp-service-prod.ol.epicgames.com (XMPP chat)
  3. Matchmaking: fortnite-matchmaking-public-service-prod11.ol.epicgames.com
  4. Asset CDN: cdn1.unrealengine.com, download.epicgames.com
  5. Telemetry: tracking.epicgames.com, analytics endpoints
  6. Geo-DNS server selection: Resolves to the nearest regional game server IP

How DNS Chains Add Latency

Each domain may involve a CNAME chain:

fortnite-matchmaking-prod.ol.epicgames.com
  → CNAME: matchmaking.fortnite.aws.epicgames.com
    → CNAME: d3abc123.cloudfront.net
      → A record: 54.192.x.x

Each CNAME hop is a sequential DNS query. With 3 hops and a 30ms DNS server, that's 90ms per domain. Multiply by 5–8 domains at launch = 450–720ms best case. If any lookup times out (default 2–5 second timeout), the chain stalls.

Quick wins for DNS:
  • Use a fast local DNS resolver: Cloudflare (1.1.1.1) or Google (8.8.8.8) typically resolve in <10ms.
  • Pre-warm DNS cache: nslookup fortnite-matchmaking-public-service-prod11.ol.epicgames.com before launching.
  • Disable DNS-over-HTTPS in Windows if your router already provides fast DNS.
  • Set shorter DNS timeout: netsh interface ipv4 set dnsservers "Ethernet" static 1.1.1.1 primary

# Find slow DNS queries (>100ms)

  -T fields -e dns.qry.name -e dns.time

9 ISP Throttling Detection

How ISPs Throttle Gaming Traffic

ISPs use several methods to selectively degrade specific traffic types:

MethodHow It WorksDetection Difficulty
Deep Packet Inspection (DPI) Examines packet payloads to identify application protocols. Can fingerprint game traffic by packet size patterns and port ranges. Hard to detect from packet captures alone (throttling looks like congestion)
Port-based throttling Deprioritizes or rate-limits traffic on known game ports (e.g., UDP 9000–9100) Moderate: compare performance on game ports vs. other ports
Time-of-day throttling Reduced priority during peak hours (6pm–11pm) Easy: capture at different times and compare
Volume-based throttling After X GB transferred, all traffic is deprioritized Easy: performance degrades after threshold

The VPN Test (Gold Standard)

The most reliable detection method: run the same game with and without a VPN. A VPN encrypts all traffic, making DPI and port-based throttling impossible. If your connection improves by >20% (lower ping, less loss) with a VPN, your ISP is almost certainly throttling gaming traffic. The VPN adds its own overhead (~5–15ms), so an improvement despite that overhead is strong evidence.

Packet Capture Indicators of Throttling

  • Selective loss on game ports: UDP 9000–9100 shows 3% loss while TCP 443 (HTTPS) shows 0% loss. This is unnatural—real congestion affects all traffic equally.
  • Consistent RTT ceiling: Game server RTT never drops below a fixed floor (e.g., always 60ms+) despite the server being geographically close. Traceroute shows the added latency at a specific hop (ISP's traffic shaper).
  • Traffic shaping saw-tooth: Throughput ramps up, hits a ceiling, drops to zero for a moment, ramps again. This is a token bucket rate limiter pattern.
  • TTL anomalies: Some ISPs inject packets with modified TTL values. Compare TTL from your game server across different times—inconsistency suggests middlebox interference.
tshark filters for throttling detection: 
# Compare loss rates between game traffic and HTTPS
# Step 1: Count game packets and look for gaps
tshark -r capture.pcapng -Y "udp.port >= 9000 and udp.port <= 9100" \
  -q -z io,stat,10

# Step 2: Count HTTPS retransmissions (should be low if ISP isn't throttling HTTPS)
tshark -r capture.pcapng -Y "tcp.port == 443 and tcp.analysis.retransmission" \
  -q -z io,stat,10

# Step 3: Check for TTL inconsistencies (middlebox detection)
tshark -r capture.pcapng -Y "ip.dst == GAME_SERVER_IP" \
  -T fields -e frame.time_epoch -e ip.ttl | sort -t$'\t' -k2 -n | uniq -f1 -c

# Step 4: Compare with VPN capture (encrypted = no throttling)
# If game RTT drops by >20% with VPN, throttling is confirmed

# Step 5: Traceroute to find the throttling hop
# tracert -d GAME_SERVER_IP

Quick Health Check: One-Command Assessment

Run this after capturing a Fortnite session: 
# Full network health report from a capture

# 1. Packet rate per second (should be stable ~30 from server)
tshark -r fortnite.pcapng -Y "udp.port >= 9000 and udp.port <= 9100 and ip.src == SERVER_IP" \
  -q -z io,stat,1

# 2. Jitter analysis (frame.time_delta stats)
tshark -r fortnite.pcapng -Y "udp.port >= 9000 and udp.port <= 9100 and ip.src == SERVER_IP" \
  -T fields -e frame.time_delta | sort -n | tail -20
# The tail shows your worst jitter spikes

# 3. TCP retransmissions (for matchmaking/auth traffic)
tshark -r fortnite.pcapng -Y "tcp.analysis.retransmission" -q -z conv,tcp

# 4. DNS resolution times
tshark -r fortnite.pcapng -Y "dns.time > 0.05" \
  -T fields -e dns.qry.name -e dns.time

# 5. Packet size distribution
tshark -r fortnite.pcapng -Y "udp.port >= 9000 and udp.port <= 9100" \
  -q -z io,stat,0,"MIN(frame.len)frame.len","AVG(frame.len)frame.len","MAX(frame.len)frame.len"
